# Wireframing

## Overview

This skill generates production-ready wireframes as self-contained HTML files, accompanied by a living specification document (`SPEC.md`) that captures all product decisions. It follows a structured conversation flow: understand the idea → clarify requirements → generate wireframes. Output uses a consistent visual language optimized for developer handoff.

**Key outputs:**
- `SPEC.md` — Single specification file, updated progressively throughout the session
- `wireframes/*.html` — Self-contained wireframe files
- `wireframes/index.html` — Workbench documentation (for multi-page projects)

The SPEC.md is designed to hand off to the **staging phase** (separate skill) where wireframes become a working React frontend with sample data.

## Conversation Flow

### Phase 1: Understand

When the user describes what they want to build, gather context:

1. **What is it?** - App type, purpose, target users
2. **What pages/screens?** - List the key views needed
3. **What's the platform?** - Web, mobile app, or both
4. **Any references?** - Existing products, styles, or inspirations

Ask clarifying questions if the request is vague. Don't assume - confirm.

### Phase 2: Structure

Before generating, confirm the plan:

```
Here's what I'll wireframe:

**Product:** [Name/description]
**Platform:** [Web / Mobile / Both]

**Pages:**
1. [Page name] - [brief purpose]
2. [Page name] - [brief purpose]
3. ...

Does this look right, or should I adjust anything?
```

Wait for confirmation before proceeding.

**→ Create SPEC.md** after user confirms. Populate sections 1-3 (Product Overview, Features, Information Architecture) with what's known so far.

### Phase 3: Generate

Create wireframes as HTML files. Each file should be self-contained with embedded Tailwind CSS.

**→ Update SPEC.md** after each wireframe:
- Add entry to Wireframe Index (section 8)
- Add any new components to Component Inventory (section 6)
- Add any discovered data entities to section 7
- Update changelog

**File naming:** Use kebab-case descriptive names
- `home.html`, `dashboard.html`, `user-profile.html`
- For mobile: `mobile-home.html`, `mobile-checkout.html`

**Output location:** Create a `wireframes/` directory in the user's project, or ask where they'd like files saved.

## SPEC.md — The Living Specification

The SPEC.md file is the **single source of truth** for everything decided during wireframing. Its sections mirror the workbench `index.html` structure. Additional sections can be added as needed for specific projects.

### When to Update SPEC.md

| Trigger | What to Update |
|---------|----------------|
| After Phase 1 (Understand) | Create file with Product Summary |
| After Phase 2 (Structure) | Add Core Features, User Flows, Sitemap |
| After each wireframe | Add to Pages section |
| When discussing requirements | Add/update UI Requirements |
| When user requests changes | Update affected sections, add to Changelog |

## Visual Language

All wireframes must follow these constraints exactly:

### Core Rules

| Element | Specification |
|---------|--------------|
| Borders | 1.5px solid black (use `.border-wire` class) |
| Corners | `rounded-md` for all elements (except avatars: `rounded-full`) |
| Colors | Black, white, and grays only. No brand colors. |
| Typography | System fonts only. No custom fonts. |
| Shadows | Simple offset only: `box-shadow: 3px 3px 0 black` (no blur) |
| Grid | 8px base unit, 16px for larger spacing |
| Text hierarchy | Use `font-light` / `font-medium` / `font-bold` - not gray colors |

### Required CSS

Every wireframe file must include this style block:

```html
<style>
  /* 1.5px border utilities */
  .border-wire { border: 1.5px solid black; }
  .border-b-wire { border-bottom: 1.5px solid black; }
  .border-t-wire { border-top: 1.5px solid black; }
  .border-r-wire { border-right: 1.5px solid black; }
  .border-l-wire { border-left: 1.5px solid black; }
  .divide-y-wire > * + * { border-top: 1.5px solid black; }

  /* Simple drop shadow (no blur) */
  .shadow-wire { box-shadow: 3px 3px 0 black; }
</style>
```

### Component Naming

Use shadcn/Radix naming conventions for easy developer handoff:

- `Button` (primary, secondary, link variants)
- `Input`, `Textarea`, `Select`
- `Checkbox`, `Radio`, `Switch`
- `Dialog`, `Sheet`, `Alert`
- `Card`, `Avatar`, `Badge`
- `Tabs`, `Accordion`
- `Table`, `Pagination`
- `DropdownMenu`, `Command`
- `Toast`, `Tooltip`

### Mobile Components

For mobile wireframes, also use:

- `StatusBar` (iOS/Android)
- `NavigationBar` (header)
- `BottomTabBar`
- `ActionSheet`, `BottomSheet`
- `FloatingActionButton`
- `DeviceFrame` (iPhone, Android, iPad)

## HTML Template

Use this base structure for all web wireframes. Wireframes use a **fixed-width container** (1024px) and are not responsive to browser width.

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>[Page Name] | Wireframe</title>
  <script src="https://cdn.tailwindcss.com"></script>
  <script>
    tailwind.config = {
      theme: {
        extend: {
          fontFamily: {
            sans: ['system-ui', '-apple-system', 'BlinkMacSystemFont', 'Segoe UI', 'Roboto', 'sans-serif'],
          },
        },
      },
    }
  </script>
  <style>
    .border-wire { border: 1.5px solid black; }
    .border-b-wire { border-bottom: 1.5px solid black; }
    .border-t-wire { border-top: 1.5px solid black; }
    .border-r-wire { border-right: 1.5px solid black; }
    .border-l-wire { border-left: 1.5px solid black; }
    .divide-y-wire > * + * { border-top: 1.5px solid black; }
    .shadow-wire { box-shadow: 3px 3px 0 black; }
  </style>
</head>
<body class="bg-gray-100 min-h-screen font-sans py-8">

  <!-- Fixed-width wireframe container -->
  <div class="w-[1024px] mx-auto">
    <div class="bg-white border-wire rounded-md overflow-hidden">

      <!-- Wireframe content here -->

    </div>
  </div>

</body>
</html>
```

## Examples

### Simple Request

**User:** "Wireframe a login page"

**Response flow:**
1. Confirm: "I'll create a login page wireframe. Should this be for web or mobile? Any specific requirements like social login or forgot password?"
2. After confirmation, create `SPEC.md` with basic product info
3. Generate `login.html` wireframe
4. Update SPEC.md with wireframe entry and components used

### Complex Request

**User:** "Design an e-commerce checkout flow"

**Response flow:**
1. Clarify: "For the checkout flow, I'm thinking: Cart → Shipping → Payment → Confirmation. Web, mobile, or both?"
2. Confirm the page list
3. Create `SPEC.md` with full product overview, features, and screen inventory
4. Generate wireframes: `checkout-cart.html`, `checkout-shipping.html`, etc.
5. Update SPEC.md progressively:
   - Add User Flow for checkout process
   - Document data entities (Cart, CartItem, Address, PaymentMethod, Order)
   - Build component inventory as screens are created
6. Create workbench `index.html` to tie them together
7. Complete staging handoff checklist in SPEC.md

## Key Principles

1. **Clarity over decoration** - Wireframes communicate structure, not aesthetics
2. **Consistency** - Same components should look identical across all pages
3. **Developer-ready** - Use real component names and patterns that map to implementation
4. **Self-contained** - Each HTML file works independently, no external dependencies except Tailwind CDN
5. **Iterate through conversation** - Users can request changes, additions, or variations
6. **Progressive documentation** - SPEC.md is updated continuously, not just at the end
7. **Staging-ready** - Document data entities and component props for smooth handoff

## Staging Handoff

When wireframing is complete, SPEC.md should contain everything needed for the **staging phase** (separate `workbench-staging` skill):

1. **Component Inventory** with props — so staging knows what React components to build
2. **Data Entities** with fields — so staging can generate TypeScript types and sample data
3. **User Flows** — so staging can wire up navigation and state
4. **UI Requirements** — so staging knows responsive breakpoints and interaction patterns

The staging skill will:
- Read SPEC.md to understand what to build
- Generate `types.ts` from Data Entities
- Generate `sample-data.json` from entity examples
- Create React components matching Component Inventory
- Build pages matching Wireframe Index
